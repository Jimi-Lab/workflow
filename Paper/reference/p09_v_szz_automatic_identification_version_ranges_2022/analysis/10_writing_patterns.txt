# 10 Writing Patterns for Reuse

Abstract Pattern:
- Operational dependency on NVD/CVE affected-version claims.
- Reliability gap in public metadata.
- Method mismatch in existing SZZ-based refinement.
- Proposed vulnerability-specific tracing method.
- Manual annotation plus quantitative comparison.
- Explicit limitation/repairable future direction.

Introduction Pattern:
- Start from security impact and public vulnerability infrastructure.
- Tie the problem to real industrial workflows such as SCA and remediation SLAs.
- Use one concrete CVE version-history change to make metadata error tangible.
- Use a second case to show algorithmic failure, not just database noise.
- Convert cases into research questions and contributions.

Method Pattern:
- Make assumptions explicit: vulnerable code presence, inducing/fixing boundary, version tags as release identifiers.
- Split the pipeline into SZZ adaptation, duplicate-change detection, and version-tag inference.
- For each stage, state the failure mode it prevents.

Evaluation Pattern:
- Establish ground truth scarcity first, then justify manual annotation.
- Explain exclusion criteria before presenting results.
- Separate VIC identification from affected-version refinement.
- Discuss false negatives as more security-critical than false positives.

Related Work Pattern:
- Separate vulnerability analysis/management from SZZ evaluation/application.
- Position the work as crossing the two: vulnerability metadata refinement through SZZ lineage analysis.

Agent Prompt Reuse:
- When writing our paper, ask the agent to extract: operational consequence, metadata unreliability, baseline assumption mismatch, method stage, and evidence table.
