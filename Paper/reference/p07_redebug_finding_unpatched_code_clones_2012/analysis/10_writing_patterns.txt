Best Structural Moves:
- Start from a concrete patch/CVE example before abstract system design.
- Frame the method as a deliberate design point rather than universally best.
- Use scale, language coverage, and false-detection cost as the motivation for a conservative approach.
- Separate algorithm description from operational workflow.
- Use concrete bug-confirmation counts to connect system output to security impact.

Best Transition Moves:
- Patch example -> unpatched code clone definition.
- Prior clone detection -> practical limitations at OS scale.
- Syntax-based compromise -> scalable workflow.
- Large-scale results -> security implications.
- Discussion -> limitations and comparison with parser-heavy clone detectors.

Best Contribution Wording Pattern:
"We trade semantic completeness for a scalable, language-agnostic, low-noise detector that can be used in day-to-day patch workflows."

Best Method Overview Pattern:
Show two flows: source preprocessing/build database and patch query/report generation.

Best Experiment Framing Pattern:
Use operational questions instead of only metric names: how fast can a database be built, how fast can patches be queried, how many clones persist, and how many are real bugs.

Useful Phrases to Paraphrase Structurally:
- "unique design point"
- "may find fewer clones, but gains scale"
- "unpatched code clones"
- "OS-distribution scale"

Patterns to Avoid:
- Avoid presenting syntax matching as semantic vulnerability verification.
- Avoid using original performance claims as local artifact claims.
- Avoid citing table values until table repair is complete.
