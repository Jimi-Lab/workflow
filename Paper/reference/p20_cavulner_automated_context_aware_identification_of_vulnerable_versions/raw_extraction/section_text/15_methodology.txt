6.2 Static Analysis for Vulnerable Version Identification
Static analysis approaches primarily comprise version-controlled commit tracing and code similarity
matching. The former identifies vulnerability-inducing commits through backward traversal from fix
commits, designating intermediate versions between inducing and fix commits as vulnerable. Typical
algorithms include the SZZ [39] algorithm and its variants AG-SZZ [24], MA-SZZ [14], RA-SZZ [32],
and V-SZZ [9]. Sliwerski et al. [39] proposed a technique known as Sliwerski-Zimmermann-Zeller

(SZZ for short) that uses the diff command provided by Version Control Systems to identify lines
of code modified by changes that fixed bugs. By tracing code history, the SZZ technique uses the
annotate command in VCS to identify changes that introduced errors. Many researchers have
improved SZZ to enhance the accuracy of identifying error-inducing commits. Kim et al. [ 24]
proposed the AG-SZZ method, which uses annotation graphs to obtain more precise results,
addressing the deficiency of non-semantic lines in the original SZZ algorithm. Costa et al. [ 14]
introduced the MA-SZZ method, which excludes meta-changes from all candidate bug-inducing
commits, i.e., commits that do not change the source code. To further reduce false positives, Neto et
al. proposed two versions of RA-SZZ [32], based on RefDiff [38] and Refactoring Miner [44], which
consider the impact of refactored lines of code in the original SZZ algorithm. Bao et al. [9] proposed
the V-SZZ method, which leverages line mapping algorithms to identify the earliest commit that
modified the vulnerable lines and then considers these commits to be the vulnerability-inducing
commits. Rosa et al. [37] used an oracle to evaluate the performance of existing SZZ and its variants.
Many works also identify vulnerable versions based on code similarity matching. ReDebug [21]
uses a fast, syntax-based method that can scale to code bases the size of operating system distribu-
tions written in many different languages. MVP [46] designed a novel slicing method based on patch
code, extracting vulnerability and repair signatures from vulnerable functions and their patched
counterparts. VUDDY [23] reduces the number of signature comparisons by utilizing function-level
granularity and length filtering techniques, proposing a scalable method to detect vulnerable code
clones. VCCFinder [34], based on data collected from code repositories, trained an SVM classifier
to tag suspicious commits. Similarly, VulPecker [28], based on a collected database of vulnerability
patches and instances of vulnerable code, as well as an automatic selection of code similarity
algorithms effective for specific vulnerabilities, solved the problem of source code vulnerability
detection. VulDeePecker [49] improved the accuracy of VulPecker through deep learning. All these
efforts target the C/C++ ecosystem, while VERJava [42] targets the Java ecosystem, which employs
a two-stage approach to systematically assess vulnerable versions.
The emergence of LLMs has brought new solutions for security research. For example, Li et
al. introduce IRIS [ 27], a neuro-symbolic approach combining LLMs with static analysis. The
LLM infers taint-flow specifications for a project and guides vulnerability detection. Fakih et al.
present LLM4CVE [18], an iterative pipeline that uses LLMs to automatically generate patches for
vulnerable functions. In terms of identifying vulnerable versions, Tang et al. propose LLM4SZZ [43]
to enhance the classic SZZ algorithm by using an LLM to identify bug-inducing commits. However,
vulnerabilities differ from bugs. Without vulnerability specific context and accurate tracing of
inducing commits, LLM4SZZ falls short on vulnerable version identification. Our approach is
specifically designed to address this issue.
