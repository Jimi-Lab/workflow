4 Evaluation
Our implementation leverages the LangChain framework [3] for systematic pipeline construction.
To mitigate stochastic noise in LLM-generated responses and reduce logical bias, we set the
temperature to zero for LLMs. Our approach is implemented in about 8,500 lines of Python code in
total. All experiments are conducted on a machine equipped with an AMD EPYC 9654 CPU, 32
GB of RAM, and running Ubuntu 20.04. We leverage API calls to interact with LLM services. We
systematically evaluate CaVulner by addressing the following three research questions (RQs):
RQ1. Effectiveness Evaluation. How accurate is CaVulner in identifying vulnerable versions?
How does the performance of CaVulner compare to state-of-the-art approaches?
RQ2. Ablation Study. What is the contribution of each component of CaVulner to its overall
effectiveness? How effective is CaVulner when applied with different LLMs?
RQ3. Resource Consumption. What are the costs associated with LLMs?
4.1 Dataset
CVE Dataset. We leveraged the dataset compiled by Ponta et al. [35], who manually collected and
curated vulnerabilities in open-source software along with their corresponding fixing commits.
This dataset was also utilized in the studies by V-SZZ [9] and VERJava [42]. The dataset of V-SZZ
covered various types of vulnerabilities. VERJava selected the TOP6 projects with respect to the
number of CVEs having available patches. Additionally, it included Liferay-portal, which contains
the highest number of CVEs in NVD. The CVEs in the VERJava dataset span a broad period of each
project’s lifecycle, represent various vulnerability types, and involve patches of varying sizes.
Our study specifically targeted CVEs from this collection that affect open-source projects hosted
on GitHub. To construct our dataset, we merged the CVEs used by V-SZZ and VERJava, then
