Problem:
- 1 Introduction Identifying the commit that introduced a bug, known as the Bug-Inducing Commit (BIC), is a foundational task in software maintenance.
- For two decades, the dominant BIC identification approach has been the SZZ algorithm [32] and its variants [7, 8, 19, 24, 35, 36], which trace deleted (and modified) code in a bug-fixing commit (BFC) via git blame to identify commits that last modified those lines.
- Presumably, one or more of those commits are the BIC.
- While SZZ operates on an implicit assumption that bug-inducing and bug-fixing commits modify the same code elements [42], this assumption does not always hold: prior work has shown that over 13% of BICs share no files with their corresponding bug-fixing commits, and 17% of bug fixes contain only added lines where blame cannot be applied [23].
- As the git blamemechanism constrains the search space to commits that directly touched the fixed lines, it misses BICs introduced through earlier code that was subsequently refactored [ 24, 25], or bugs in related code where the fix addresses the issue by adding new functionality.
