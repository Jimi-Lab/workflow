II. BACKGROUND AND RELATED WORK
In this section, we ﬁrst provide an overview of the original
SZZ algorithm, outlining its background and highlighting some
limitations related to ghost commits. Next, we study the variants
of the SZZ algorithm that have been proposed to improve its
precision and recall. Lastly, we discuss the application and
signiﬁcance of the SZZ algorithm and its variants within the
Software Engineering research community.
We focus on the eight SZZ algorithms that are considered
by Rosa et al. [28]: B-SZZ [1], AG-SZZ [2], DJ-SZZ [15],
L-SZZ [16], R-SZZ [16], MA-SZZ [12], RA-SZZ [13], RA-
SZZ* [14], as well as some implementation variants: PyDriller
(SZZ@PYD) [33], SZZ Unleashed [34], and OpenSZZ [35].
A. The Original SZZ Algorithm & Ghost Commits
SZZ, the name given to the algorithm initially introduced
by ´sliwerski, Zimmermann, and Zeller [1], was developed to
identify changes that induce bugs. In our forthcoming discus-
sion, we will follow Rosa et al. [28] and refer to this original
SZZ algorithm as B-SZZ. B-SZZ follows a core structure that
is shared with its subsequent variants: (1) identifying bug-
ﬁxing commits, followed by a ﬁltering process utilizing two
conﬁdence levels: syntactic and semantic; (2) mapping bug-
ﬁxing commits to their corresponding potential bug-inducing
commits; and (3) eliminating potential bug-inducing commits
that are unlikely to be the cause of bugs.
1) Identifying Bug-ﬁxing Commits:
In the initial stage,
B-SZZ endeavors to establish a relationship between a bug-
ﬁxing commit in the version control system (VCS) and the cor-
responding bug reports in the issue-tracking system (ITS). To
accomplish this, B-SZZ employs two independent conﬁdence
levels: a syntactic level, which infers links between a VCS log
and a bug report, and a semantic level, which validates these
links through bug report data.
For most software systems, there is no mandatory or default
link between bug reports in the ITS and the VCS. Thus, devel-
opers may forget to add such links. Rezk et al. [17] deﬁne the
missing bug-ﬁxing commits that SZZ implementations fail to
detect, where links were omitted, as Ghost Commit 0 (GC 0).
Herbold et al. [36] also ﬁnd that half of the bug-ﬁxing commits
identiﬁed by SZZ are incorrect due to the mislabeled issues
in the ITS (where developers incorrectly label issues as bugs,
while some are merely improvements, etc.) [37]. Herbold et al.
also ﬁnd that SZZ misses about one-ﬁfth of the bug-ﬁxing
commits.
This paper will not study the limitations of bug-ﬁxing commit
identiﬁcation, as: (1) previous work [38], [39], [40] has already
provided a comprehensive analysis and proposed tools to re-
cover GC 0’s missing links, and (2) the Linux kernel repository
requires developers to use the “Fixes:” tag along with the ﬁrst
12 characters of the SHA-1 ID when a patch addresses a bug
in a speciﬁc commit. Therefore, we can collect a “developer-
informed” Linux kernel dataset using the VCS without reliance
on the ITS.

been identiﬁed, B-SZZ proceeds to the second step, where it

B-SZZ ﬁrst employs CVS’s diff command.2 This command
is analogous to git show, as it identiﬁes the lines that have
been deleted or modiﬁed by a commit. Then, B-SZZ invokes
the annotate command (another CVS command, similar to git
blame, but that does not identify the corresponding lines in
earlier revisions) for each line to determine the most recent
commit that deleted or modiﬁed the line prior to the ﬁx. When
there are multiple removed or modiﬁed lines, multiple commits
can be mapped as potential bug-inducing commits. This paper
focuses on issues in the mapping step.
Tools such as CVS annotate and git blame cannot map bug-
ﬁxing commits that do not modify or remove lines to bug-
inducing commits. This situation is deﬁned as Mapping Ghost
type 1 (MG 1) by Rezk et al. (To give a more meaningful
naming for MG 1, we name it Remove Mapping Ghost in
this paper.) For example, in Fig. 2, bug-ﬁxing commit 7fdbc5f3
does not remove any lines and only added two lines in all the
changes.
Similarly, if a bug-inducing commit does not add lines, it
cannot be tracked by using blame on subsequently removed
lines, as blame can only track previously existing lines. This
situation is deﬁned as Mapping Ghost type 2 (MG 2) by
Rezk et al. (To give a more meaningful naming for MG 2, we
name it Add Mapping Ghost in this paper.) For instance, in
Fig. 3, bug-inducing commit 5a858e74 does not add any lines
and only removes two lines in all the changes.
3) Filtering: In the ﬁnal stage, B-SZZ discards the potential
bug-inducing commits that cannot have caused the bug, such
as those committed to the VCS after the bug was reported.

Subsequent studies have proposed various ﬁltering methods,
including Content Filters [2], [41] and Suspiciousness Filters

for a given bug-ﬁxing commit, Rezk et al. refer to the bug-ﬁxing

We do not further examine the limitations of the ﬁltering step
in the SZZ algorithms for two reasons: (1) The ﬁltering step
heavily depends on data from the ITS, but we bypass any ITS
by using the tags found directly in the Linux kernel commits;
(2) The ﬁltering process aims to improve precision by eliminat-
ing commits from the potential bug-ﬁxing commits, while our
focus is on enhancing the recall of the existing SZZ algorithm,
i.e., adding potentially correct bug-inducing commits that were
missing.
B. Improvements for B-SZZ
Despite the numerous proposals aimed at enhancing the ac-
curacy of the SZZ algorithm [2], [12], [13], [14], [15], [16], the
examination of bug-inducing commits not detected by SZZ is
still in its infancy, and only limited efforts have been made to
improve the recall [2], [17], [19].
Kim et al. [2] found that B-SZZ may incorrectly label bug-
inducing commits due to considering formatting or cosmetic
changes. To address this, they proposed AG-SZZ, which ig-
nores such changes during the mapping process, improving the
accuracy of locating bug-introducing commits. Moreover, as
B-SZZ employs the CVS annotate command for mapping lines
removed in bug-ﬁxing commits to recent revisions, it cannot
pinpoint the corresponding lines in earlier revisions. Kim et al.
observed that such mappings might lead to inaccurately linked
code segments. AG-SZZ, therefore, utilizes annotation graphs
[42] to map line numbers across revisions, ensuring that both
the starting and mapped lines belong to functions with the same
name, improving the accuracy of the mapping process.
Williams et al. [15] propose DJ-SZZ to address the lim-
itations of annotation graphs which cannot track individual
lines through extensive modiﬁcations by using a line-number
mapping approach [43]. Furthermore, they used DiffJ [44] to
exclude non-executable changes.
Davies et al. [16] propose L-SZZ and R-SZZ, which improve
on the AG-SZZ algorithm by selecting only one bug-inducing
commit from all the candidate commits. L-SZZ selects the
commit that changes the most lines. R-SZZ selects the most
recent one.
Da Costa et al. [12] found that AG-SZZ identiﬁes meta-
changes (i.e., branch changes, merge changes, and property
changes) as bug-inducing commits. AG-SZZ includes these
commits as the annotation graph is unable to map the code
between meta-changes and their preceding changes. To address
this issue, Da Costa et al. propose MA-SZZ that connects
all nodes in the annotation graph to their prior changes. This
enhancement ensures that MA-SZZ does not identify meta-
changes as bug-inducing commits.
Neto et al. [13] found that refactoring operations, which do
not modify a program’s behavior, impact the SZZ results. They
extended MA-SZZ as two variants, RA-SZZ [13] and RA-SZZ*
[14], to reduce the set of false positives reported by MA-SZZ.
RA-SZZ and RA-SZZ* use RefDiff [29] and Refactoring Miner
[30], respectively, to exclude refactoring operations from bug-
inducing commit candidates, but both algorithms only work on
Java ﬁles.
PyDriller (SZZ@PYD) [33], SZZ Unleashed [34], and Open-
SZZ [35] are three open-source implementations of SZZ al-
gorithms. PyDriller is a Python framework that analyzes Git
repositories by extracting information about commits, diffs,
and other information. It also implements B-SZZ and ignores
C-style and Python-style comment lines. SZZ Unleashed uses
line-number mapping without DiffJ, which allows it to work
on non-Java ﬁles. OpenSZZ implements B-SZZ by using the
git blame command.
To improve the recall of the SZZ algorithm, Sahal et al. [19]
propose a syntax-based variation, A-SZZ, to address the case
of bug-ﬁxing commits that do not have removed lines (Remove
Mapping Ghost). This algorithm traces changes for all lines
within the code blocks that contain the additions. Rezk et al.
[17] enhance the A-SZZ approach by incorporating context-
aware data ﬂow analysis, which includes mitigation strategies
for speciﬁc programming scenarios such as null checks, cre-
ation of new entities, method overrides, logging operations, and
class expansions. These strategies use data ﬂow analysis on
identiﬁers in added code lines, and then use B-SZZ to map the
lines within these identiﬁers’ data ﬂow paths.
Table I presents the datasets that have been used for evalu-
ating these algorithms, detailing the dataset type, the number
of projects, and the bug-ﬁxing commit count. In most cases,
the ground truth is labeled by researchers without specialized
domain knowledge. MA-SZZ uses an evaluation framework
when there is no ground-truth label available between the bug-
ﬁxing commit and the bug-inducing commit. Speciﬁcally, it
uses an automatic metric with three criteria to evaluate SZZ:
earliest bug appearance, future impact of a change, and realism
of bug introduction [12]. The only dataset labeled by develop-
ers contains, on average, only 1.2 commits per project, which
may not sufﬁciently represent the entirety of issues arising in
the development process of each project. This underlines the
signiﬁcance of collecting a dataset annotated by developers,
encapsulating a range of problems that could emerge during a
project’s lifespan.
C. SZZ in SE Research

SZZ plays in software engineering research [28], [45], [46].
SZZ is used as the ﬁrst step in many studies, especially in
the area of defect prediction. Many defect prediction studies
use the SZZ algorithm to build datasets to train and evaluate
their models [5], [6], [11], [20], [21], [22], [23], [24]. Herbold
et al. [36] found that as of August 2019, 8 out of 15 of the public
datasets used for defect prediction used B-SZZ to construct their

III. CONSTRUCTING A DATASET OF BUG-INDUCING COMMITS
INFORMED BY DEVELOPERS
Evaluating the accuracy of the SZZ algorithm and its vari-
ants requires a dataset of bug-ﬁxing commits accompanied in
each case by the set of commits that introduced the ﬁxed bug.
´sliwerski, Zimmermann, and Zeller [1] proposed experiments
based on ECLIPSE and MOZILLA in which bug-ﬁxing com-
mits were chosen based on their references to a bug tracker;
no evaluation was made of the accuracy of the corresponding
collected bug-inducing commits. Subsequent SZZ variants were
mostly evaluated using small, manually-validated datasets, as
shown in Table I.
A. The Dataset of Rosa et al.
In 2021, Rosa et al. [28] proposed a new methodology to
build a dataset for evaluating SZZ variants based on the obser-
vation that the log messages of some commits both contain a
statement that the commit is ﬁxing a bug and explicitly identify
the commit that introduced the bug that is being ﬁxed. For
example, they have found the message “THRIFT-4513: ﬁx bug
in comparator introduced by e58f75d” in the commit a8a97bd
in the project apache/thrift.5 The presence of such messages
raises the possibility of creating a large-scale dataset fully auto-
matically, relying on the expertise of project developers to map
bug-ﬁxing commits to the corresponding set of bug-introducing
commits. Nevertheless, they did not identify a standardized
means of providing bug-ﬁxing and bug-introducing commit in-
formation neither across projects nor within projects. Thus, they
proposed to mine such information in commit logs using natural
language processing (NLP) techniques, followed by manual
veriﬁcation. Using their approach, they created a dataset of
1,930 bug-ﬁxing commits from 1,625 software projects written
in a variety of programming languages. Using the resulting
dataset, they compare the precision, recall, and F-measure of
nine SZZ variants, and study the impact of dropping reported
bug-inducing commits that postdate the associated entry in the
bug tracker.
While the NLP-based approach of Rosa et al. [28] made
it possible to obtain a large dataset, exploiting developer
knowledge, researchers have identiﬁed some weaknesses in
their results. The dataset remains small, containing 1,930 bug-
ﬁxing commits from 19,603,736 commits collected on GitHub,
amounting to 0.001% of the total commits collected. While not
all commits on GitHub are ﬁxes or introduce bugs, previous
studies [49], [50] have highlighted that 55% to 95% of the
total cost of software development is devoted to maintenance.
Therefore, it is likely that the 1,930 commits collected in the
dataset is a vast underestimation of the actual number of bug
ﬁxes. And as there are on average only 1.2 commits per project,
the collected commits may not be representative of the full
scope of issues that arise in each project’s development.
B. A Dataset Extracted From the Development History of the
Linux kernel
The Linux kernel was ﬁrst released in 1991 and has grown to
23 MLOC as of the release of Linux v6.0 in October 2022. The
Linux kernel has been developed using git since April 2005.
Prior to October 2013, Linux kernel developers sporadically
included in their commit log messages comments beginning
with Fixes: followed by a reference to Bugzilla or some other
URL. Starting with the commit 629c66a22c216 from security
researcher Kees Cook on October 24, 2013, the Linux kernel
developer community has mostly settled on a strict format
for bug-ﬁx information. This format consists of one or more
lines starting with the keyword Fixes:, each followed by the
commit id of the bug-introducing commit, followed by the
ﬁrst line of the commit log message of the bug-introducing
commit in parentheses. This format is encouraged by the Linux
kernel documentation7 and by the Linux kernel “linting” tool
checkpatch.8
