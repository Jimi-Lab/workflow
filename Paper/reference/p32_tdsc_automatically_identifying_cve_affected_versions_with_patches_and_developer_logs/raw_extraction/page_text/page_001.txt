Automatically Identifying CVE Affected Versions
With Patches and Developer Logs
Yongzhong He , Yiming Wang , Sencun Zhu ,W e iW a n g, Yunjia Zhang, Qiang Li,a n dA i m i nY u
Abstract—While vulnerability databases are important sources
of information for software security, it is known that information in
these databases is inconsistent. How to rectify these incorrect data
is a challenging issue. In this article, we employ developer logs and
patches to automatically identify vulnerable source code versions
that each CVE really affects. Our tool organizes all versions of a
piece of software into a version tree, and identiﬁes the ﬁrst vulner-
able version, and the last vulnerable versions in the version tree
trunk and branches. For evaluation, we took Linux Kernel as the
case study and quantiﬁed the error rate of the vulnerable versions
reported by the NVD. The total number of vulnerable Linux Kernel
versions reported by the NVD was 43,727 (as of September 2020),
of which the total number of false positives reached 2,497 and the
total number of false negatives reached 9,330, accounting for 5.7%
and 21.34%, respectively. In addition, we compare our tool with two
vulnerability detection tools and show that our tool could achieve
high detection accuracy.
Index Terms—Developer logs, NVD, patch, versions, vulnerab-
ility.

Security professionals rely on various sources of information
to inform them of new security issues. An important source
is the vulnerability database. Public vulnerability databases in-
clude CVE, NVD, SecurityFocus, Redhat Bugzilla, ExploitDB,
OpenWall, SecurityTracker, etc., but we focus on the most
widely used and authoritative database: National Vulnerability
Database (NVD)[3] in this work. The database is maintained by
the U.S. government, committed to accurately record all known
vulnerabilities, and effectively as an industry standard. For ex-
ample, business security services (such as Hakiri[4],S n y k[5]
and SourceClear [6]) and open-source security tools (such as
BudlerAudit [7], OW ASP OSSIndex[8] and dependency check-
ing [9]) rely on NVD vulnerability information to run effectively.
Researchers [10], [12] use the NVD as the core data source
to reveal all aspects of vulnerability discovery and recovery
process. Aksu et al.[13] rely on the information of the NVD to
generate attack graph automatically. However, is the information
from the NVD database accurate? Indeed, previous work has
pointed out the inaccuracy of NVD information. For example,
Dong et al.[14] reported the inconsistent vulnerable software
versions of the NVD are highly prevalent: only 59.82% of the
vulnerability CVE summaries strictly match the standardized
NVD entries, and the inconsistency level increases over time.
Anwar et al.[15] discovered the inconsistent or incomplete data
of the NVD and showed the possible impact of these data prob-
lems. Other work[16], [17], [18] also investigated some types
of NVD data quality problems. The above work indicated that
the known affected software conﬁgurationof NVD (introduced
in SectionII-A) was indeed inaccurate. However, the previous
work mostly pointed out the inaccuracy of the NVD, with
the following question largely remaining unanswered:which
vulnerable versions are exactly affected by a CVE-ID?
