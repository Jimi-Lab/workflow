Problem:
- Abstract—While vulnerability databases are important sources of information for software security, it is known that information in these databases is inconsistent.
- In this article, we employ developer logs and patches to automatically identify vulnerable source code versions that each CVE really affects.
- Our tool organizes all versions of a piece of software into a version tree, and identiﬁes the ﬁrst vulner- able version, and the last vulnerable versions in the version tree trunk and branches.
- For evaluation, we took Linux Kernel as the case study and quantiﬁed the error rate of the vulnerable versions reported by the NVD.
- The total number of vulnerable Linux Kernel versions reported by the NVD was 43,727 (as of September 2020), of which the total number of false positives reached 2,497 and the total number of false negatives reached 9,330, accounting for 5.7% and 21.34%, respectively.
Gap:
- However, is the information from the NVD database accurate?
- However, the previous work mostly pointed out the inaccuracy of the NVD, with the following question largely remaining unanswered:which vulnerable versions are exactly affected by a CVE-ID?
- To ﬁll this gap, we aim to propose a practical solution to justify the incorrect versions reported by the NVD.
- For a given CVE, none of the existing vulnerability detection methods can guarantee that the vulnerability can be accurately identiﬁed in all released versions, so just relying on these methods is not enough.

Evaluation Setup:
- results from Vulnerable Version Finder with the Known Affected Software Conﬁgurations of the NVD, and outputs the justiﬁed affected versions and the accuracy of the NVD.
- We collected Linux kernel vulnerabilities released from January 2003 to September 2020 from the NVD, which resulted in 2,576 CVE -IDs in total.
- Accuracy Evaluation We scan 1,367 CVE-IDs of Linux kernel against its 3,086 open-source versions to detect all vulnerable versions in the three directions (R1, R2 and R3).
- For our dataset, we manually verify the reported boundary of vulnerable versions in the version tree for each CVE-ID.
- However, for FNs, since there are a great number of software versions(we do not regard the versions reported by NVD as the ground truth because it has many errors, as presented in the next subsection), we cannot directly report the FN rate or recall.
- Therefore, for R1 and R2, we sample 100 CVE-IDs for manual inspection to calculate the recall of our tool.
- As for R3, we determine the lower bound of the recall.

Limitations or Scope Boundaries:
- Sometimes patches may not be implemented correctly, which means that the software versions are vulnerable even if they are patched.
- We plan to study how to identify incorrect(incomplete or wrong) patches in the future work.
- For future research, it will be interesting to solve the above limitations, and explore whether partial matching and repatching can be extended for binary code vulnerability detection.
