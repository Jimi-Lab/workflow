CaVulner: Automated Context-Aware Identification of
Vulnerable Versions
ANONYMOUS AUTHOR(S)
Modern software systems increasingly rely on third-party open-source components, which now constitute
the majority of code in typical applications. While this reuse accelerates development, it also exposes systems
to significant security risks, as known vulnerabilities in reused components can propagate downstream. To
assess such risks, maintainers and Software Composition Analysis (SCA) tools depend on public vulnerability
databases, such as the National Vulnerability Database (NVD) and GitHub Advisories, which specify affected
version ranges. However, recent studies have shown that these databases often contain inaccurate or incomplete
version metadata, leading to false negatives that leave systems unpatched or false positives that trigger
unnecessary and potentially disruptive upgrades. This issue is amplified by the rapid growth in reported
vulnerabilities and the adoption of large language models (LLMs), which accelerate vulnerability detection but
shift pressure to the slower downstream tasks of confirmation and auditing.
In this paper, we propose CaVulner, an automated, LLM-assisted framework for context-aware identification
of vulnerable versions. To localize vulnerable code, CaVulner first narrows candidate methods using commit
change patterns and CVE/CWE metadata, then guides the LLM to extract the root cause through fine-grained
reasoning over pre- and post-fix code. To infer affected versions, CaVulner traces vulnerability-inducing
commits using git blame, identifies semantically duplicated commits via hunk hash comparison, and normalizes
their associated tags into standardized version ranges. We evaluated CaVulner on 43 real-world Java open
source projects that included 176 CVEs and 8,730 versions. The results show that CaVulner significantly
outperforms state-of-the-art approaches, including V-SZZ, VERJava, and LLM4SZZ. It achieves an accuracy
gain of at least 35.8% on CVE level. At the version level, F1-score increases by at least 23.94%, and the cost of
using the LLM per vulnerability is only around $0.06. It offers a scalable and semantically grounded solution
for supply chain security.
1
Introduction
Modern software development heavily relies on third-party open-source software (OSS). The World
Economic Forum highlights that OSS makes up as much as 70% to 90% of the code in software
systems [10]. While the reuse of OSS significantly improves development efficiency and reduces
costs, it also introduces substantial security risks. For instance, a report [6] by Black Duck reveals
that 86% of codebases contain components with known vulnerabilities.
The number of known software vulnerabilities has been increasing rapidly in recent years,
which extensively influenced the ecosystem. To manage this growing threat, organizations such
as the National Vulnerability Database (NVD) [31] and GitHub Advisory Database [2] curate
critical information on disclosed vulnerabilities, to support the wide usage of software supply
chain tools like Software Composition Analysis (SCA), to assess potential risks. In this case,
SCA tools rely heavily on the accuracy of vulnerability data, especially the correct vulnerable
versions, for automated mappings of security risks in downstream user projects. However, multiple
studies [9, 16, 33, 42] have shown that these vulnerability records often contain inconsistencies
or errors, which can seriously mislead maintenance decisions. Moreover, as the integration of
large language models (LLMs) has significantly accelerated the discovery of new vulnerabilities,
the downstream tasks, like vulnerability confirmation and auditing, face increasing pressure to
keep pace, and more automated solutions with less human effort, especially on vulnerable version
identification, are urgently needed.
To precisely identify vulnerable versions in OSS, researchers have proposed a variety of solutions
broadly categorized into dynamic analysis and static analysis. Dynamic approaches [26, 45, 47, 48]
typically employ fuzzing to confirm whether a vulnerability can be triggered. While dynamic meth-
ods produce no false positives, they inherently suffer from false negatives due to the fundamental

earlier code. For example, Nikolaos et al. [5] found that the average lifetime of a vulnerability is

the latest comprehensive empirical study [13] on this topic. After evaluating 12 state-of-the-art
tools (including both tracing and matching-based methods) on over 1,000 vulnerabilities, the study
revealed that no existing tool achieves an accuracy above 45.0%, largely due to their inability to
handle complex patch patterns and lack of deep semantic understanding.
In summary, existing approaches face two key challenges. Challenge 1: How to distinguish
security fixes from non-security fixes within a fix commit and accurately identify the vulnerability-
introducing code. Challenge 2: How to effectively trace back to earlier vulnerable versions con-
taining equivalent vulnerable code, given that traditional code similarity analysis and AST-based
line mapping algorithms often fail due to code evolution over time.
To overcome these challenges, we propose a novel approach named CaVulner, which integrates
multi-source context and leverages the capabilities of LLMs for vulnerability analysis and code
understanding to identify vulnerable versions within the Java ecosystem. For Challenge 1, we
develop a two-stage LLM-based module that leverages contextual cues while minimizing noise.
The first stage focuses on coarse-grained localization by narrowing down candidate methods based
on fix commit change patterns, such as whether a method contains only deletions, only additions,
or both, and by incorporating relevant vulnerability information from CVE descriptions and CWE
classifications. This guides the LLM to focus on methods most likely to contain the root cause. The
second stage performs fine-grained reasoning by prompting the LLM to analyze semantic differences
between the pre- and post-fix code within the selected method to identify the precise code fragment
responsible for the vulnerability. For Challenge 2, we trace the origin of each vulnerable code
block using a combination of git blame and LLM reasoning. We handle complexities such as cherry-
picks, merges, and duplicated commits by computing hunk hashes to detect semantically identical
changes across different commit hashes. All tags pointing to the inducing commits are collected,
and the difference between these and the fix commit tags is computed. The resulting tag set is then
normalized into standardized version ranges, representing the vulnerable versions.
Our experiments demonstrate the effectiveness of CaVulner on 176 real-world CVEs spanning
43 Java open-source projects, covering a total of 8,730 versions. Compared to state-of-the-art
approaches, CaVulner improves at least 35.80% in accuracy at the CVE level. At the version level,
precision increases by at least 10.40%, recall by 14.59% and F1-score by 23.94%. However, the cost of
using the LLM remains low. On average, the LLM cost per vulnerability is only around $0.06.

Preliminaries
In this section, we introduce the concepts of vulnerability-inducing commit, and a motivating
example.
2.1
Vulnerability-Inducing Commit
The concept of Vulnerability-Inducing Commit (VIC) is conceptually derived from the well-
established and widely studied notion of Bug-Inducing Commit (BIC) [39], which is formally
defined as the commit in a version control system where a software defect is first introduced.
Building on this foundational premise, a considerable number of researchers have dedicated ef-
forts to proposing and refining improved techniques for identifying BICs, most notably through
the development and refinement of various SZZ-based algorithms [14, 24, 32]. The original SZZ
algorithm fundamentally operates by systematically tracing backward through the version control
history starting from a known bug-fixing commit, with the objective of precisely identifying and
isolating the specific earlier commits that were responsible for introducing the particular lines
of code that were later modified or deleted in that fix, a process typically and effectively using
built-in version control annotation or blame mechanisms. These subsequent variants of the SZZ
methodology significantly enhance the original method by addressing common issues such as
noise, inaccurate line mapping, and commit granularity.
Furthermore, with the recent and rapid emergence of powerful large language models, a growing
body of recent studies has actively explored their potential integration into and enhancement
of traditional SZZ-based workflows. For example, Tang et al. introduced LLM4SZZ [43], which
leverages the semantic capabilities of language models to more accurately detect bug-inducing
commits based on patch content and commit messages. By understanding the contextual meaning
of code changes and commit descriptions, LLM4SZZ reduces false positives and identify extra
commits that all baselines cannot.
Given the success of SZZ in the bug localization domain, it has subsequently been adapted for
vulnerability analysis, particularly in identifying vulnerability-inducing commits and inferring
vulnerability-affecting version ranges. One notable and direct adaptation of this methodology is
exemplified by V-SZZ [9], which operates by attempting to systematically trace backwards from a
known vulnerability-fixing commit in order to identify and pinpoint the specific earlier commit
that originally introduced the security flaw, primarily through the analysis of lines of code that
were deleted within the fix patch. However, this direct adaptation introduces significant limitations.
Specifically, V-SZZ assumes all deleted lines are vulnerable and fails to address fix commits that
only contain added lines.

bugs and vulnerabilities that are often not well accounted for in these approaches. For instance, while
bugs are frequently introduced in recent commits and can often be detected within a short temporal

The motivating example is illustrated in Figure 1, which examines CVE-2018-1000406, an arbitrary
file-write vulnerability caused by Jenkins’s failure to properly normalize and constrain user input.
The correct range of affected versions for this vulnerability is [1.312, 1.658] ∪[2.0, 2.145]. However,
all three tools report the range as [1.362, 1.658] ∪[2.0, 2.145], incorrectly identifying 1.362 as the
lower bound. This discrepancy indicates a shared weakness across these tools in accurately tracing
back to the true inducing commit.
Each tool failed for different reasons:
• For V-SZZ, it identifies vulnerability-inducing commits by treating every deleted line in the
fix commit as vulnerable and tracing them back using the git blame command, aided by an
AST-based line mapping to handle refactorings. In this case, it incorrectly attributes the commit
e96c8f8 (corresponding to version 1.362) as the inducing commit. However, deeper semantic
inspection reveals that the actual vulnerable logic was introduced in a descendant commit of
e96c8f8, which V-SZZ fails to reach due to it only tracing the vulnerable codes identified in the
pre-patch commit, while neglecting the syntactic changes of these code during the backward
tracing. Additionally, V-SZZ is unable to handle fixes that contain only added lines, rendering it
ineffective in such scenarios.
• For VERJava, on the other hand, it uses code similarity to match vulnerable code patterns
across versions. While this helps to detect structurally similar code, it struggles when semantic
equivalence exists without syntactic similarity, particularly when code evolves significantly
across versions. As a result, it fails to recognize vulnerable logic in earlier versions, leading to
false negatives.
• For LLM4SZZ, it employs a large language model to analyze commit messages and code diffs
to identify bug-inducing commits. Despite its improved performance for bugs, it still fails to
identify the correct vulnerability-inducing commit in this case, again misattributing e96c8f8 as
the root cause. This failure highlights a key limitation: without contextual information such as
CVE descriptions and a deeper understanding of both pre-fix and post-fix code, the model lacks
the necessary reasoning to detect subtle security vulnerabilities. Moreover, like other SZZ-based
approaches, LLM4SZZ is not equipped to perform extended historical tracing, which is often
required to uncover long-lived vulnerabilities.
These findings collectively underscore the inadequacy of current approaches in reliably iden-
tifying vulnerability-inducing commits. Accurate vulnerable version identification demands not
only precise syntactic tracing but also deep semantic understanding of code changes and security
context.
3
Approach

leverages the capabilities of LLMs in vulnerability analysis and code understanding to identify
vulnerable versions.

blocks most likely responsible for the vulnerability, based on insights from multi-source context
such as CVE descriptions and code context. Next, it traces these code blocks back to the specific
commits that introduced the vulnerability, leveraging LLMs to reason about semantic changes
that traditional syntax-based tools may miss. Finally, by analyzing both the inducing and fixing

3.1
LLM-based Vulnerable Code Block Identification
The first step progressively narrows down the location of vulnerable code through two stages.
It begins with coarse-grained method localization, identifying methods likely associated with
the vulnerability based on contextual signals such as CVE descriptions, commit messages, and
surrounding code. Building on this, the module then performs fine-grained analysis to isolate the
specific code blocks within those methods that directly correspond to the vulnerable behavior. By
combining these two stages, the module achieves both breadth and precision in identifying the
code regions most relevant to the vulnerability.
3.1.1
Coarse-grained Vulnerable Method Localization. Directly feeding raw fix commit diffs to
the LLM often leads to hallucinations because the model may struggle to distinguish the actual
remediation logic from irrelevant code churn. To mitigate this issue, this submodule serves as a
pre-filter to identify methods strictly relevant to the vulnerability and thereby reduces the search
space for the subsequent fine-grained analysis. However, effectively filtering these methods presents
a challenge as developers frequently interleave security patches with non-security maintenance
such as refactoring, formatting, or feature updates.
To address this, we introduce Method Change Type (MCT) as a semantic abstraction mechanism.
Unlike raw line-level diffs, MCT categorizes how a method is modified, establishing a correlation
between the nature of the repair and the vulnerability type. For instance, fixing a Missing Input
Validation vulnerability typically involves adding conditional checks, whereas remediating an
Information Exposure vulnerability involves deletions. By explicitly defining these patterns, we
guide the LLM to focus on methods whose modification signatures align with the expected repair
strategy for the specific vulnerability.

and a standardized description is provided, detailing the characteristics of the vulnerability.
• CWE (Common Weakness Enumeration). CWE is a community-maintained classification

common weaknesses in code, design, or logic (e.g., root causes of vulnerabilities). It reveals the

Guided Prompt Construction. We employ a guided prompting mechanism that integrates role
assignment, multi-source context and an expert analysis framework to direct the LLM’s reasoning.
• Role Assignment. The LLM is assigned the role of a senior Java vulnerability analysis expert
and is tasked with performing logical inference based on the provided multi-source context.
• Multi-source Context. This consists of method change types, CVE ID, CVE description, CWE
ID and CWE name.
• Expert Analysis Framework. We prompt the LLM to consider how different MCTs may be
related to the remediation of vulnerabilities (e.g., line additions might indicate the introduction
of new security checks). The model is guided to integrate multi-source data with its reasoning
capabilities in order to identify the methods most associated with vulnerability and explain the
rationale behind its conclusions.
• Output. The LLM outputs the signatures of the methods it has identified as vulnerable, which
serve as part of the input to the subsequent fine-grained vulnerable code block identification
module.
3.1.2
Fine-grained Vulnerable Code Block Identification. This submodule builds on the vulnerable
candidate methods identified in the previous coarse-grained submodule by using the LLM to perform
a fine-grained analysis. The goal is to identify vulnerable code blocks within these methods.
Specifically, for each vulnerable candidate method, we extract its complete code content before
and after the fix and generate a diff between them. Compared to analyzing the full diff of a fix
commit, which may contain much non-security content, this method-level organization significantly
reduces the processing burden on the LLM and improves the accuracy of its analysis. Additionally,
we provide the LLM with two supplementary inputs: the CWE description and the commit message
for the fix. The CWE description highlights common vulnerability patterns and root causes, and the
commit message explains the developer’s intention behind the fix. Together, these inputs enable
more accurate vulnerability analysis.
Prompt Construction and LLM Reasoning Process. We design prompts for the LLM that
include the following information and guide it through a specific reasoning process.
• Input Information. This includes the complete source code of the vulnerable candidate method
in both its pre-fix and post-fix versions, along with the corresponding code differences. Addi-
tionally, it contains the CWE description and the fix commit description.

3.3
Vulnerable Version Determination
In the context of version control systems (e.g., GitHub), tags are commonly used to denote versions.
Once the LLM identifies the vulnerability-inducing commit (VIC) corresponding to a given fix
commit, we can determine the set of vulnerable tags by examining the tags that include the VIC
but not the fix commit.
However, in the real world, repositories often maintain multiple active branches (e.g., main,
release-1.0 and release-2.0), and developers often propagate both VICs and fix commits across these
branches through merges, cherry-picks, or manual copying. These operations result in duplicate
commits, which are commits that introduce the same changes but have different commit IDs.
Failure to identify all instances of duplicate fix commits or VICs can result in inaccurate reporting
of vulnerable tags, leading to an underestimation or overestimation of their scope.
Duplicate Commit Identification. We identify duplicate commits by comparing the content
of their patches. Specifically, we compute a hash for each hunk in every commit. For a given
commit, we identify all other commits that share the same hunk hashes. If the set of hunk hashes
in the candidate commit is equal to, or a subset of, that of another commit, we regard them as
duplicates. This enables us to track semantically equivalent changes across branches, ensuring a
more comprehensive identification of commits that introduce or fix vulnerabilities.
Vulnerable Tags Determination. Given a CVE, along with its associated fix commits and
VICs, we analyse the git tags in the project’s repository to identify the vulnerable tags. For each fix
commit and its duplicates, we collect all tags that contain these commits, labeled 𝑇𝑓. Similarly, for
each VIC and its duplicates, we gather all the corresponding tags, 𝑇𝑖. The set of vulnerable tags 𝑇𝑣
is then those tags that belong to 𝑇𝑖but not to 𝑇𝑓:
𝑇𝑣= { 𝑡| 𝑡∈𝑇𝑖∧𝑡∉𝑇𝑓}
This approach enables accurate identification of affected tags, even in complex branching sce-
narios involving cherry-picks or manual copying.

these vulnerable lines through the version history. For each traced version, we examined whether
the previous version contained the vulnerability. If not, we consider the current version to be the

fixed. In total, this process resulted in a high-quality ground truth that includes 17,436 vulnerable

4.3
RQ1. Effectiveness Evaluation
We select three representative state-of-the-art methods for identifying vulnerable versions in the
Java ecosystem as our baselines: V-SZZ [9], which is based on commit tracing; VERJava [42],
which leverages code similarity; and LLM4SZZ [43], which utilizes LLMs for vulnerability versions
identification.
V-SZZ traces all deleted lines in a fix commit back to their origin using git blame, treating
them as vulnerability-inducing lines. To improve the accuracy of locating vulnerability-inducing
commits (VICs), it also uses an AST-based line mapping algorithm. However, V-SZZ cannot handle
cases where the fix commit contains no deleted lines, and the AST-based line mapping is not
always accurate. VERJava addresses this limitation by supporting fix commits without deleted lines.
It computes vulnerable versions at both the function level and the entire patch level, and then
synthesizes the results to determine the final set of vulnerable versions. Since VERJava requires
comparing every version, the computational cost is very high. As a result, the authors evaluated
only a subset of versions. In contrast, to ensure the accuracy of our evaluation, we evaluate all
versions of each project. For example, the VERJava study only assessed 13 versions of the Liferay
Portal project, while we evaluated all 233 versions. LLM4SZZ employs rank-based identification
and context-enhanced identification to handle different types of fixing commits.
Table 2 and 3 present the results of CaVulner and baselines in identifying vulnerable versions in
the CVE level and version level, respectively. Both LLM4SZZ and CaVulner use DeepSeek V3 as the
large language model. According to the evaluation metrics, the criteria for CVE level statistics are
particularly strict: if even a single predicted vulnerable version does not match the ground truth,
the result cannot be counted as true. As a result, the accuracy at the CVE level is generally lower
than precision and recall at the version level.
Despite the stringent requirements for CVE-level accuracy, Table 2 reveals that CaVulner achieves
the highest overall accuracy of 73.30%, outperforming LLM4SZZ (33.52%), V-SZZ (37.50%) and
VERJava (20.45%). These results highlight the effectiveness of CaVulner in accurately identifying
complete vulnerable version sets for each CVE.
We examined the 47 failure cases where CaVulner failed, aiming to clarify the underlying causes
of these inaccuracies. Specifically, 8 cases erroneously included non-vulnerable early versions,
while 39 failed to cover the complete vulnerable range. Further analysis reveals that 27 failures
stemmed from misjudged semantic equivalence, while 14 cases resulted from errors in identifying
duplicated commits, and the remaining 6 were caused by incorrect initial code localization.
At the version level, where predictions are evaluated at a finer granularity, Table 3 demonstrates
that CaVulner again achieves superior performance across all metrics. It achieves the highest
precision (98.14%), recall (83.15%), and F1-score (90.02%), indicating that it can reliably identify
most vulnerable versions while introducing few false positives. In comparison, LLM4SZZ achieves
a lower recall (47.20%) and F1-score (61.38%), suggesting its vulnerability identification is more
conservative and incomplete. Meanwhile, VERJava, although showing slightly better recall (68.56%)
than V-SZZ (51.21%), suffers from a high false positive rate, as reflected by its lower precision
(63.78%) and F1-score (66.08%).
Notably, CaVulner achieves 100% precision in several projects such as jackson-databind, jenkins,
spring-security, and struts, indicating its ability to avoid introducing any irrelevant versions during
prediction. Even in large-scale projects like jenkins and jackson-databind, which contain thousands
of versions, CaVulner maintains consistently high accuracy and recall.

CVE-level accuracy requires an exact match of the entire vulnerable version range, any deviation
in identifying the VIC results in a failure for that CVE. By replacing the AST algorithm with our
LLM-based semantic tracing, CaVulner significantly improves the accuracy to 73.30%. This absolute
gain of 7.39% demonstrates that the LLM-based approach is superior in correctly pinpointing the
exact VIC. Unlike the AST method, which may be disrupted by complex code refactorings and
fail to identify the true origin, the LLM’s semantic understanding enables CaVulner to correctly
determine the start of the vulnerability lifecycle in substantially more cases, thereby securing the
correctness of the final version sets.
At the version level, CaVulner_diff achieves a high precision (94.11%) but suffers from poor recall
(58.12%) and a moderate F1-score (71.86%). CaVulner_ast improves substantially on recall (80.23%)
and F1-score (88.72%) and achieves high precision (99.22%). However, both its recall and F1-score
remain lower than those of CaVulner. This demonstrates that although AST-based tracing can
boost precision, integrating LLM-driven VIC tracing is essential to maximize recall and overall F1
performance.

5
Discussion
Threats to Validity. In constructing the ground truth, we manually identified the vulnerable
versions for each CVE, which may introduce potential inaccuracies. To mitigate this threat, three
authors with cybersecurity expertise, including one Ph.D. student and two master’s students,
independently examined each CVE based on their domain knowledge and publicly available
information. Discrepancies were resolved through collaborative discussion until consensus was
reached. On average, six CVEs were processed per day, and the entire process took approximately
one month. To capture the complexity of real-world vulnerabilities, our dataset incorporates
diverse data qualities. Among the 176 selected CVEs, 58.6% contain low-quality commits with
irrelevant messages, 9.9% involve complex fixes that modify more than five files, and 5.7% lack
CWE information. Such heterogeneity suggests that CaVulner possesses a significant degree of
tolerance toward low-quality input data.
LLM Capability and Overhead. Previous approaches to vulnerable version identification were
limited in their ability to understand vulnerability semantics and code logic. The emergence of
large language models (LLMs) has addressed this limitation by enabling more accurate semantic
reasoning. However, LLMs are not always precise and may still produce incorrect results. As LLM
technology continues to evolve, its effectiveness in vulnerability analysis is expected to improve
further. Although LLM-based identification requires more time compared to traditional methods,
the improvement in accuracy makes the additional computational cost acceptable.
Other Ecosystems. This study focuses on the Java ecosystem for three reasons. Java is the
dominant language in enterprise-level applications, and its vulnerabilities can have far-reaching
consequences. For example, Log4Shell vulnerability remains one of the most critical flaws ever
discovered [15]. Moreover, the first author has worked extensively on Java vulnerability research,
including zero-day discovery, and possesses deep practical expertise with Java codebases and patch
processes. This experience substantially reduces annotation errors when constructing the ground
truth of vulnerable versions. Finally, to ensure accurate and efficient extraction of method change
patterns from fix commits, we employ a Java AST parsing library. Extending vulnerable-version
identification to other ecosystems such as C, C++, and Python represents an important direction
for future work.

vulnerability by attempting to trigger its underlying abnormal program behavior under controlled

For example, SemFuzz [47] proposes and develops a methodology fundamentally rooted in
program semantics, which is precisely architected for the automated synthesis of functional Proof-
of-Concept (PoC) exploit code, then repurposes the synthetically generated PoCs as seed inputs
to direct a subsequent, systematic fuzzing campaign, with the objective of triggering and thereby
uncovering latent vulnerabilities embedded within the target software system. PatchFuzz [45]
begins its operation by first identifying the precise Git commits that are documented as fixes
for specific security flaws, then proceeds to extract the associated PoC for each fix, and finally
synthesizes this collected information with the semantic and syntactic details of the patch diff
to guide focused fuzzing on the modified code regions. TransferFuzz [26] adopts a trace-guided
fuzzing approach wherein it meticulously records and archives the execution traces from a known
vulnerable version of the program, subsequently reusing and adapting these historical traces to
precisely steer and prioritize its fuzzing campaigns when targeting newer, patched, or derivative
versions, thereby systematically determining whether the propagation of reused code components
continues to expose the same underlying vulnerability. DeltaFuzz [48] performs an extraction of
version-difference information and strategically utilizes this to prioritize and guide the generation of
fuzz inputs along the repair boundary, with the ultimate objective of verifying whether subsequent
releases or later versions of the software remain susceptible to the original security flaw.
In addition, Directed Greybox Fuzzing (DGF) plays a crucial role in vulnerable version identifica-
tion by steering the fuzzer towards specific target sites, such as potentially vulnerable functions.
AFLGo [11] integrates a distance-based metric into the energy scheduling of the fuzzer, prioritizing
inputs that execute paths closer to the target vulnerability, thus facilitating the efficient repro-
duction of known CVEs across different versions. Building on this, Hawkeye [12] optimizes the
direction mechanism by incorporating static analysis of the call graph and function-level similar-
ity, addressing the issue of trace explosion in long-path vulnerability verification. More recently,
WindRanger [17] introduces a deviation-based directed fuzzing approach specifically designed for
patch analysis. It identifies deviation basic blocks between the vulnerable and patched versions and
directs the fuzzer to explore these divergences, making it highly effective for verifying whether a
specific version contains the unpatched logic. To further improve efficiency in large-scale systems,
Beacon [20] utilizes static analysis to compute reachability preconditions and aggressively prunes
execution paths that cannot reach the target vulnerability, thereby significantly accelerating the val-
idation process. Similarly, SelectFuzz [30] enables selective fuzzing of specific code regions within
complex binaries by isolating the execution of relevant functions, allowing for precise verification
of vulnerabilities in targeted components without the overhead of whole-system execution.
While dynamic analysis methods produce no false positives, they inherently and unavoidably
suffer from the problem of false negatives. This limitation is fundamentally rooted in the stochastic
nature of fuzzing techniques, which often struggle to achieve sufficient code coverage or satisfy
complex path constraints required to trigger deeply hidden vulnerabilities. Furthermore, estab-
lishing, configuring, and maintaining isolated and functionally complete testing environments
for every distinct library version or dependency encountered in a large-scale system proves to be
impractical, thereby imposing significant scalability constraints on the broader application of this
methodology.
6.2

matching. The former identifies vulnerability-inducing commits through backward traversal from fix
commits, designating intermediate versions between inducing and fix commits as vulnerable. Typical
algorithms include the SZZ [39] algorithm and its variants AG-SZZ [24], MA-SZZ [14], RA-SZZ [32],
and V-SZZ [9]. Sliwerski et al. [39] proposed a technique known as Sliwerski-Zimmermann-Zeller

(SZZ for short) that uses the diff command provided by Version Control Systems to identify lines
of code modified by changes that fixed bugs. By tracing code history, the SZZ technique uses the
annotate command in VCS to identify changes that introduced errors. Many researchers have
improved SZZ to enhance the accuracy of identifying error-inducing commits. Kim et al. [24]
proposed the AG-SZZ method, which uses annotation graphs to obtain more precise results,
addressing the deficiency of non-semantic lines in the original SZZ algorithm. Costa et al. [14]
introduced the MA-SZZ method, which excludes meta-changes from all candidate bug-inducing
commits, i.e., commits that do not change the source code. To further reduce false positives, Neto et
al. proposed two versions of RA-SZZ [32], based on RefDiff [38] and Refactoring Miner [44], which
consider the impact of refactored lines of code in the original SZZ algorithm. Bao et al. [9] proposed
the V-SZZ method, which leverages line mapping algorithms to identify the earliest commit that
modified the vulnerable lines and then considers these commits to be the vulnerability-inducing
commits. Rosa et al. [37] used an oracle to evaluate the performance of existing SZZ and its variants.
Many works also identify vulnerable versions based on code similarity matching. ReDebug [21]
uses a fast, syntax-based method that can scale to code bases the size of operating system distribu-
tions written in many different languages. MVP [46] designed a novel slicing method based on patch
code, extracting vulnerability and repair signatures from vulnerable functions and their patched
counterparts. VUDDY [23] reduces the number of signature comparisons by utilizing function-level
granularity and length filtering techniques, proposing a scalable method to detect vulnerable code
clones. VCCFinder [34], based on data collected from code repositories, trained an SVM classifier
to tag suspicious commits. Similarly, VulPecker [28], based on a collected database of vulnerability
patches and instances of vulnerable code, as well as an automatic selection of code similarity
algorithms effective for specific vulnerabilities, solved the problem of source code vulnerability
detection. VulDeePecker [49] improved the accuracy of VulPecker through deep learning. All these
efforts target the C/C++ ecosystem, while VERJava [42] targets the Java ecosystem, which employs
a two-stage approach to systematically assess vulnerable versions.
The emergence of LLMs has brought new solutions for security research. For example, Li et
al. introduce IRIS [27], a neuro-symbolic approach combining LLMs with static analysis. The
LLM infers taint-flow specifications for a project and guides vulnerability detection. Fakih et al.
present LLM4CVE [18], an iterative pipeline that uses LLMs to automatically generate patches for
vulnerable functions. In terms of identifying vulnerable versions, Tang et al. propose LLM4SZZ [43]
to enhance the classic SZZ algorithm by using an LLM to identify bug-inducing commits. However,
vulnerabilities differ from bugs. Without vulnerability specific context and accurate tracing of
inducing commits, LLM4SZZ falls short on vulnerable version identification. Our approach is
specifically designed to address this issue.
7
Conclusion
In this paper, we presented CaVulner, a novel approach for the precise identification of vulnerable
versions in Java open source projects. CaVulner integrates multiple sources of information, including
method level change patterns, CVE and CWE metadata, and patch diffs, with the semantic reasoning
power of large language models. It localizes vulnerable code blocks, traces the commits that
introduced those vulnerabilities and derives an accurate set of affected version tags by deduplicating
hunk hashes and normalizing version labels. Our evaluation on 176 real-world CVEs from 43 projects
covering 8,730 versions demonstrates that CaVulner outperforms state-of-the-art approaches,
including V-SZZ, VERJava, and LLM4SZZ in accuracy, precision, recall, and F1-score. While this
study focuses on the Java ecosystem, we will adapt CaVulner to additional language ecosystems
such as C, C++, and Python in future work.
